# ワークフロー設計ガイド

あなたはこのワークフロー設計に従って、すべてのタスクを体系的に進める。

## 原則 1: Planモードを基本とする

**いつ適用するか:** タスクが3ステップ以上 or アーキテクチャに関わる場合は**必ず**Planモードで開始する。

進め方:
1. `EnterPlanMode` ツールを使ってPlanモードに入る
2. 実装前に詳細な仕様を `tasks/todo.md` に書く
3. チェック可能な項目として計画を整理し、ユーザーに確認を取る
4. うまくいかなくなったら無理に進めず、すぐ立ち止まって再計画する

---

## 原則 2: サブエージェント戦略

**いつ適用するか:** リサーチ・調査・並列分析が必要な場面。

進め方:
- メインのコンテキストウィンドウをクリーンに保つためAgentツールを積極活用する
- サブエージェント1つにつき1タスクを割り当てる
- 複雑な問題には複数のサブエージェントを並列で投入する
- 調査結果はメインスレッドに集約してから実装に進む

---

## 原則 3: 自己改善ループ

**いつ適用するか:** ユーザーから修正を受けた後、必ず実行する。

進め方:
1. `tasks/lessons.md` を読んで既存のパターンを確認する
2. 今回の修正内容をパターンとして追記する
3. 「なぜそのミスが起きたか」「次回どうすれば防げるか」を記録する
4. セッション開始時にプロジェクト関連のlessonsをレビューする

---

## 原則 4: 完了前に必ず検証する

**いつ適用するか:** タスクを「完了」とマークする直前。

チェックリスト:
- [ ] 動作を証明できるか（テスト実行 or ブラウザで確認）
- [ ] mainブランチとの差分に問題ないか
- [ ] 「スタッフエンジニアはこれを承認するか？」と自問した
- [ ] ログを確認し、正しく動作することを示せるか

---

## 原則 5: エレガントさを追求する（バランスよく）

**いつ適用するか:** 重要な変更を実装する前。シンプルな修正はスキップしてよい。

進め方:
1. 実装前に「もっとエレガントな方法はないか？」と一度立ち止まる
2. ハック的な修正に感じたら「今知っていることをすべて踏まえてエレガントな解決策を実装する」
3. 自分の作業を提示する前に自問自答する
4. 過剰設計しない — 必要最低限の複雑さで解決する

---

## 原則 6: 自律的なバグ修正

**いつ適用するか:** バグレポートを受けた時。

進め方:
1. ログ・エラー・失敗しているテストを自分で読んで原因を特定する
2. 手取り足取り教えてもらわずそのまま修正する
3. ユーザーのコンテキスト切り替えをゼロにする
4. 言われなくても失敗しているCIテストを修正しに行く

---

## タスク管理フロー

```
1. tasks/todo.md に計画を書く
2. ユーザーに確認を取る
3. 実装しながら進捗をマークしていく
4. 完了前に検証チェックリストを実行する
5. tasks/todo.md にレビューセクションを追加する
6. 修正があれば tasks/lessons.md を更新する
```

---

## このスキルの使い方

`/workflow` と入力すると、このガイドがロードされる。
タスク開始時に「どの原則を適用すべきか」を判断し、それに従って進めること。

**3ステップ以上のタスクなら、まずPlanモードに入れ。** それが最初の一手だ。
